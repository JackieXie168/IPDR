Goal of Reference Libraries

 

The objective of the file sharing  library is to provide software implementers of the IPDR standards with libraries and tools to reduce construction cost, increase quality, and increase interoperability of their IPDR facilities.

 

These libraries simplifes the task of file sharing for an IPDR Producer  and IPDR Consumer. This library obviates need of request/response based communication channel between Producer and Consumer applications.

 

The libraries are available in ANSI C and Java and currently support NDM-U version 3.0

Using the C Libraries

 

The File based transfer protocol C library can be used by Producer and Consumer services.

 

A Producer creates IPDR documents (abbreviated as IPDRDocs) and places them in files, with one IPDRDoc per file.  The IPDR Producer sorts each IPDRDoc into one or more groups.   IPDRDocs will often be grouped by service type, by customer name, or by a combination of the two, though the IPDR Producer is also allowed to use any other useful policy to sort IPDRDocs into groups. These groups are referred to as subscription groups to match the terminology used in the transfer protocol. The Producer makes an entry for each Subscription Group in a Capability File, which contains the details of the Control File for that Subscription Group. IPDRDoc locations are added to these Control Files. If only the name of the document is present then the IPDRDocs will be in the same directory as the control file.

 

A Consumer reads the IPDRDocs related to a particular Subscription Group by using the Capability file, Control files and Range files. The Consumer reads the Capability file to get the location of Control files for a Subscription group. The Range file indicates the sequence numbers of Control files that a Consumer has to read. The Consumer gets the names of the Control file and reads them sequentially. It then reads the IPDRDocs indicated by the location specified in the Control file.

 

 

A sample Capability File is as shown below:

<?xml version="1.0" encoding="us-ascii" ?>

<CapabilityRsp>

    <supportedProtocolList>

        <supportedProtocolItem version=3.0 protocolMapping=”File” encoding=”XML”>

            <primitiveList>

                <primitiveItem>Pull</primitiveItem>

            </primitiveList>

            <extension>

                <groupInfoList>

                    <groupInfoItem>

                        <groupId>sm</groupId>

                        <controlFileDirectory> http://www.best.com/sm</controlFileDirectory>

                        <controlFilePrefix>sm-</controlFilePrefix>

                        <controlFileNamePolicy>NNNNNNNN</controlFileNamePolicy>

                        <controlFileSuffix>-control</controlFileSuffix>

                        <controlFileDirectoryURL>http://192.168.122.11/ipdr/sm/</controlFileDirectoryURL>

                    </groupInfoItem>

                    <groupInfoItem>

                        <groupId>voip</groupId>

                        <controlFileDirectory>file://C/IPDR/voip/</controlFileDirectory>

                        <controlFilePrefix>voip_IT1_</controlFilePrefix>

                        <controlFileNamePolicy>NNNNNN</controlFileNamePolicy>

                        <controlFileSuffix>.log</controlFileSuffix>

                    </groupInfoItem>

                </groupInfoList>

           </extension>

        </supportedProtocolItem>

    <supportedProtocolList>

<CapabilityRsp>

 

where,

groupInfoItem: refers to a SubscriptionGroup

groupId: The <groupId> element must contain the name of the subscription group described by this <groupInfoItem> element.   It should be a non-zero length string of allowable characters.  Some examples might include: “voip”, ”email-1”, and “best-hosting-co”.

 

controlFileDirectory:  <controlFileDirectory> contains the path of the directory containing the control files and the range file for the subscription group. 

 

controlFilePrefix: The <controlFilePrefix> element contains a string of allowable characters that denotes the first part of the name of all of the control files in the subscription groups.  Some possible examples include: “voip-”, “internet-access_”, and “best.com.”

 

controlFileNamePolicy: The <controlFileNamePolicy> element must contain one or more ‘N’ characters, where the number of ‘N’ characters denotes the number of digits in the sequence number embedded in the name of all control files for this subscription group.  Some examples include: “NNNNNN”, and “NNNNNNNNNN”.

 

controlFileSuffix: The <controlFileSuffix> element contains a string of allowable characters that denotes the last part of the name of all of the control files in the subscription groups.  Some possible examples include: “.log”, “-control” and “”.

controlFileDirectoryURL: The <controlFileDirectoryURL> contains the URL that points to the directory containing the control files and the range file for the subscription group.  A URL is required here so that Consumer applications are able to access the files. The URL can either be a “file:”, ”ftp:” or an “http:”.  Some examples might include:

 “file://192.168.122.11/C/IPDR/voip1/”, “http://www.best.com/voip1”.

 

Specifying a URL:

The URL of any file can be specified in three ways. These ways correspond to the protocols being used for accessing the URL:

1.       For HTTP:

The URL to a file or to a directory can be specified using the following syntax.

http://<hostname>/<directoryStructure>

e.g. For specifying the URL to a directory ipdr/Voip/ (where ipdr islying in the root directory for HTTP access) on a host specified by the name www.ipdr.com, use

http://www.ipdr.com/ipdr/Voip/

In case, IP Address is there, then specify http://<IPAddress>/ipdr/Voip/

 

2.       For FTP:

The URL to a file or to a directory can be specified, for FTP, using the following syntax.

ftp://<hostname>/<directoryStructure>

In case, the user name and password are also required, then the URL string should be:

ftp://<username>:<password>@<hostname>/<directorystructure>

e.g. So, URL for a directory ipdr/Voip/ (ipdr is lying in the ftp root directory) on ftp.ipdr.com, will be

ftp://ftp.ipdr.com/ipdr/Voip/

or for a user “ipdr” and password “ipdr”, URL will be:

ftp://ipdr:ipdr@ftp.ipdr.com/ipdr/Voip/

 

3.       For FILE:

This is the protocol that is used to access files located on local machine.

The URL to a file or directory located on the local machine is specified as :

In case of Windows:

file://localhost/ipdr/Voip/ or as file:///c:/ipdr/Voip/

In case of Linux, Solaris or HPUX:

file://localhost/home/ipdrftp/ipdr/Voip/ or file:///home/ipdrftp/ipdr/Voip/

where, ipdrftp is the user name.

Data Structures

 

Before using the API’s it is important for the developer to understand all the data structures and input parameters to various library calls.

Structure Name

Description

SubscriptionGroup

This structure contains all subscription group members namely groupId,controlFilePrefix,contrilFileDirectory,controlFileSuffix,

controlFileNamePolicy,controlFileDirectoryURL.

SubscriptionGroupList

This is used to create linked list of subscription groups.

CapabilityFile

This structure contains capabilityFileURL and subscription group linked list.

ControlFile

This structure contains control file information i.e.URL ,location ,document count ,number of versions and control file pointer .

ListGroupID

This structure is used to create linked list of groupIds of all the subscription groups.

ErrorStructure

This structure is used for error handling and contains function name in which error occurred,error message and error code.

ListErrorStructure

This is used to create linked list of error structures and is an input parameter to all the library API’s.

CapabilityXMLInfo

This structure is passed as an input parameter to parse capability xml file and is populated with the subscription group information.

Using FTP C Library for Producer Application

 

1.       Call method newListErrorStructure() to instantiate ListErrorStructure.

 

e.g. : ListErrorStructure* pErrorList = newListErrorStructure();

 

Note:The developer should always call this method in the beginning of consumer/producer flow  as pErrorlist is passed as an argument to all library API’s.

 

2.       Call readCapabilityFile() method to parse xml capability file at remote location.

CapabilityFile* readCapabilityFile(ListErrorStructure* pErrorList ,

                                                 char* capabilityFileXML);

 

Ø       pErrorList is used internally by the library for error handling.

Ø       capabilityFileXML is the URI of the  xml capability file to be parsed.

 

e.g. : CapabilityFile* capabilityFile =  readCapabilityFile(pErrorList , capabilityFileXML);

 

 

The developer should call this method otherwise when capability file URL is not available or not to be used.

 

Call newCapabilityFile() method

CapabilityFile* newCapabilityFile(pErrorList);

 

Ø       pErrorList is used internally by the library for error handling.

 

e.g. : CapabilityFile* capabilityFile =  newCapabilityFile(pErrorList );

 

3.       A SubscriptionGroup can be added to the CapabilityFile by calling addGroup() method.This is a two step process.You need to initialize the subscription group to be added.

Ø       controlFileDirectory is the control file directory on subscription group.

Ø       controlFilePrefix is the control file prefix in subscription group.

Ø       controlFileNamePolicy is the control file naming policy in subscription group.

Ø       controlFileSuffix is the control file name suffix in subscription group.

Ø       controlFileDirURL is the control file directory URL in subscription group.

 

 

 

e.g.: SubscriptionGroup* subscriptionGroup =

                                     “Voip”,

                                               “./home/John/Voip/”,

Then Call addGroup() method                                                                                         

            int addGroup(ListErrorStructure* pErrorList ,

                                  CapabilityFile* pCapabilityFile ,

                                  SubscriptionGroup* pSubscriptionGroup);

 

Ø       pErrorList used internally by library for error handling.

Ø       pCapabilityFile is the instance of  CapabilityFile structure to which the subscription group is added.

Ø       pSubscriptionGroup is the subscription group to be added.

 

e.g : result = addGroup(pErrorList, pCapabilityFile, pSubscriptionGroup);

 

Note: If the  value of result is ‘0’ then the call is successful else some error occurred.

‘result’ will be used through this document to tell the status of the call.

Appendix- Some other API’s defined in FTP C Library

 

1.       refreshControlFile() method.

This method refreshes local copy of the control file  so that updated contents can be read.

However the records are read  from the line next to the one that has been already read from the control file. This is required so that a Consumer can see the updated contents of a control file being updated by a Producer. The next getDocName() call will return the next IPDRDoc name if anything new has been added to the control file.

Note :- The Consumer will need to call this method if getDocName() method returns “NULL” as document name.This indicates that it hasn’t encountered “Version 2” string at the end of control file and might be the Producer has updated it at remote location.So to continue reading updated control file the consumer may call refreshControlFile().

 

2.       getFileStreamFromURL () method.

The library provides it as a utility function.This method retrieves file from remote URI and make a local copy at specified local path.It opens the local copy in read mode and returns the file stream to the calling application.

 

FILE* getFileStreamFromURL(ListErrorStructure* pErrorList,

                                          char* remoteURL,

                                          char* localPath)

 

Ø       pErrorList is used for error handling.

Ø       remoteURL is the URI of file to be retrieved.

Ø       localPath is the path where local copy is to be created.

 

e.g : FILE* stream = getFileStreamFromURL(pErrorList , remoteURL , localPath);

 

3.       printLastError() method.

This method can be invoked if the value of ‘result’ variable after any library call is ‘1’ or if the API’s that return pointer variable return NULL.It will print the last occurred error details with function name where it occurred and some error message.

 

e.g : if((result = <library API call>)!=0)

                  printLastError(pErrorList);

 

4.       printAllErrors() method.

This method can be called by the calling application to print all the errors that have occurred during execution of consumer or producer flow.

 

e.g : printAllErrors(pErrorList);
