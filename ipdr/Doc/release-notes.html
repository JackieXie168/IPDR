Release notes for IPDR (31 May 2005)

This document contains notes on the changes and limitations that apply to the IPDR exporter (client) and collector (server). These notes are current as of May, 31, 2005.

The most recent specification for IPDR streaming protocol can be found at http://www.ipdr.org/download-docs/protocol.html

There is also an IETF RFC for CRANE (RFC 3423), the previous version of the protocol, which is included with the sources. Some minor errata are in file rfc3423-errata.txt.

You can run the IPDR collector stand-alone. Instructions are in the "Test" class included with the (IPDR collector sources).

If you wish to unpack and build exporter from a command line, you can do the following (assume that the sources are in IPDR_distribution_2002-21-04.tar.gz, and that you're running ksh or bash and have GNU make and gcc):

gunzip -c IPDR_distribution_21042005.tar.gz | tar xf - cd IPDR_distribution_21042005 export IPDR_ROOT=`pwd` export PLATFORM=solaris # or linux for Linux machines export IPDR_PLATFORM=$PLATFORM export USE_GCC=1 cd ipdr/src/Simulators/IPDRSim make To build collector, do: cd IPDR_distribution_21042005 cd ipdr/Collector make collector The default location of the JDK is /usr/j2se, use make collector JSDK_ROOT=<another location> if it is placed elsewhere on your system. To run collector, use     make run

You can also run the collector with the following shell command (JDK location is assumed the same as above):

java -classpath ipdrapi.jar:collector.jar:jdom.jar:.:/usr/j2se/jre/lib/rt.jar org.ipdr.collector.Test <arguments>

The syntax of collector command line is as follows:

Test <debug level ([D]ebug | [I]nfo | [W]arn | [E]rror | [F]atal)> <collector config file> <CRANE legacy config file> <output type> <output file (CSV) or output prefix(IPDRXML)><IPDR XML config file (IPDR XML only)>

The parameters may only be omitted from the end of the command line, in which case the following default values will be used:
Parameter Default value
debug level [D]ebug
collector config file config.xml
CRANE legacy config file CRANEConfig.xml
output type xml
output file or output prefix IPDRXMLOutput
IPDR XML config file IPDR35XMLOutputConfig.xml

The collector accepts up to four arguments, which can be ommitted from end: configuration file, template configuration, output file, CRANE legacy configuration file. The default values for the files are, correspondingly, config.xml, templates.xml, output.txt, CRANEConfig.xml.

The following is the list of issues and limitations that apply to the software.

Common notes

  • As both Exporter and Collector may initiate connections between themselves, setting both ends of the connections to active type may cause connections to open and close franticly. Users are advised to pay attention to avoid this.

Exporter notes

  • Platforms that are currently supported by the exporter are:  SPARC/Solaris, INTEL/Linux.  Specifically, the provided code was compiled and tested on SunOS 5.8 (Solaris 8), GCC 3.3.2 and on RedHat Linux 9 (2.4.20-8), GCC 3.3.3.
  • The Exporter allows large degree of flexibility with regard to template management. However, certain limitations do apply to its session and template management capabilities. In particular, creating two templates with different fields but identical ids in different sessions of a single Exporter may cause unexpected results and even a crash.
  • Before you run IPDRSimple implementation, open IPDRSimple.c file, go to line 356 and change the makeIP(10, 1, 100, 81) expression to the IP address of your actual collector.
  • When passing document id, make sure you pass 16 bytes of data. If you pass an array which is shorter than 16 in length, garbled data will result. If you pass a longer array, only first 16 bytes will be used.
  • In order to maintain API backward compatibility, former CRANE functions were changed to macros. The following table summarizes changes that were made in order to emulate those CRANE functions for which the above change was not possible. Also check CRANE.h for specific details

  • The complete list of all Exporter parameters can be found here. These parameters are also accesible via console parameters command and by SNMP.

Collector API additions

  • public interface ConnectInfo
    The interface encapsulates connectioninformation. An instance of this interface is passed to a state machine listener whenever an Exporter connects to the Collector.
  • public interface DisconnectInfo
    The interface encapsulates disconnection information. An instance of this interface is passed to a state machine listener whenever an Exporter disconnects from the Collector and a DISCONNECT message arrives or is sent.
  • public interface FlowStartInfo
    The interface encapsulates FLOW_START message information. An instance of this interface is passed to a state machine listener whenever the Collector sends a FLOW_START message to an Exporter.
  • public interface FlowStopInfo
    The interface encapsulates FLOW_STOP message information. An instance of this interface is passed to a state machine listener whenever the Collector sends a FLOW_STOP message to an Exporter.
  • public interface SessionStartInfo
    The interface encapsulates SESSION_START message information.An instance of this interface is passed to a state machine listener whenever an Exporter sends a SESSION_START message to the Collector.
  • public interface SessionStopInfo
    The interface encapsulates SESSION_STOP message information.An instance of this interface is passed to a state machine listener whenever an Exporter sends a SESSION_STOP message to the Collector.
  • public interface ConnectionController
    The interface allows more precise control over Collector state machine. It is queried by Collector before an attempt to initiate connection is made. It is also queried by Collector before each flow is started.
  • public interface ConnectionControllerFactory -
    This interface priovides Collector with a factory for ConnectionControllers.

Collector API changes

Changes to the collector's API were made to enable recognition of the connecting exporter's internal IP address within its LAN.
When a connect message is received (by the IPDR state machine) through one of the connection threads (configured to a specific exporter) the following takes place:

  • in exporterAddressKnown method (from the IncomingConnectionHandler interface implemented by CollectorManager):
    • finds the dispaching exporter in the collector's configured exporters list.
    • finds the related connection thread in the connection thread list.
    • adds the connection thread to the exporter's connections list.
    • removes the connection thread from the collector manager connection threads list.
    • completes the initialization of the connection thread by initializaing connection configuration and exporter handler.
  • Completes initialization of IPDR state machine by creating instances of the outputter and template negotiator from their factories.

These changes are possible due to the following additions:

  • public class CollectorManager implements IncomingConnectionHandler
    New parameters:
    • protected ArrayList connectionThreads = new ArrayList() -
      Holds connection threads of all incoming/outgoing connections.
    • protected ConnectionControllerFactory connectionControllerFactory -
      Holds instance of connection controller factory.
    • private ProbingListener probingListener -
      Holds instance of probing listener.
    New methods:
    • public ConnectionConfigI exporterAddressKnown(InetAddress exportersAddress, ConnectionThread thread) -
      Holds connection threads of all incoming/outgoing connections.
    • protected ConnectionControllerFactory connectionControllerFactory -
      Tries to relate the exporter's IP to an exsiting exporterHandler.
    • public void setProbingListener(ProbingListener probingListener) -
      Sets the probing listener of the collector manager and all its registered exporter handlers.
  • public class ConnectionThread extends Thread
    New parameters:
    • protected IncomingConnectionHandler connectionHandler -
      Stores interface for handling incoming connections.
    • protected ConnectionController connectionController -
      The interface allows to control the ability of establishing flow start state in a connection.
    • private ProbingListener probingListener -
      Probing for specified sessions in a specified exporter.
    • protected OutputterFactory outputterFactory -
      Stores an instance of an Outputter factory.
    • protected TemplateNegotiatorFactory negotiatorFactory -
      Stores an instance of template negotiator factory.
    New methods:
    • public void completeThread(ExporterHandler parent, ConnectionConfigI config) -
      Completes the work done in the passive connection connection thread constructor.
    • public void setProbingListener(ProbingListener probingListener) -
      Set class parameter probingListener.
    • public void stopFlow(byte sessionId) -
      stop flow with specified session.

Additional collector changes

templates.xml is no more in use.
Enabling field negotiation in each of the collector's configured templates is done via config.xml
Under the XML element tag "TemplateConfig" each template field is identified with the element tag "FieldConfig" by the attribute tag "keyId". The enable/disable property of each template field is determined by the attribute tag "fieldUsed". Configurating "fieldUsed" with a "true"("false") value enables(disables) the field negotiation by the collector.

Exporter additions

A couple of callback function were added:

  • The following function sets the callback which is called whenever the configuration ID changes:
    IPDR_API IPDR_SetConfigurationIDChangedCallback(IPDR_ExporterHandle ExporterHandle,
    IPDR_ConfigurationIDChangedCallback Callback,void *Arg);
    The type definition of the callback function itself which is passed as the "Callback" parameter is:
    typedef void (*IPDR_ConfigurationIDChangedCallback)(unsigned char SessionID,
    unsigned short NewConfigurationID,void *Arg);
  • The ability to set a callback which will be called the first time configuration ID is set can be done via:
    IPDR_API IPDR_SetNewConfigurationIDCallback(IPDR_ExporterHandle ExporterHandle,
    IPDR_NewConfigurationIDCallback Callback,void *Arg);
    The type definition of the callback function itself which is passed as the "Callback" parameter is:
    typedef void (*IPDR_NewConfigurationIDCallback)(unsigned char SessionID,
    unsigned short* NewConfigurationID,void *Arg);

Issues fixed

  • Duplicate flag in DATA message was fixed.
  • IPDR_StopSession behavior was changed to more reasonable, that is, it now stops session always, disregarding queue drain.
  • CsvOutputter now prints out the document id in hexadecimal format.
  • IPDR_EnableKey during active session was fixed.
  • In IPDRsim, it is no longer possible to send data on a bogus (not configured) template.
  • CsvOutputter now prints config Id and duplicate flag.
  • NullOutputter was added to collector implementation.
  • Collector now respects "strict templates" parameter.
  • CsvOutputter prints value separators correctly.
  • 64-bit values are now handled correctly in both Exporter and Collector.
  • Compilation warnings were completely eliminated.
  • All C files were converted to Unix format.
  • IPDR_SendData_All and IPDR_EnableKey_All were fixed.
  • All IPDR_TYPE_* constants were removed from the code
  • C data types were defined for each XDR type
  • IPDRsimTemplates.cpp was updated with new C data types.
  • IPDR_GetCollectorInSessionList function was fixed.
  • IPDR_SetDocumentID now resets DSN.
  • ACCESS_BY_FUNCTION now handles lengths of string correctly.
  • IPDR_RemoveCollectorFromSession succeeds when a session is stopped via an API function call.
  • Changing exporter's debugging level is operational.
  • Exporter sends error to the connected collector on session delete.
  • MIB support is fixed and supported by exporter.
  • MIB command line works the same as is Command Line Interface.
  • IPDR_GetServerStatisticsAndState is implemented.
  • Exporter sets the duplicate flag on data messages it sends.

Known issues

  • Collector's configuration file, config.xml, requires a mandatory attribute, count, to be added to every Connection element. This attribute is unused and should always be set to 1.
  • While in IPDR_SendData, the exporter cannot process incoming data acknowledges. This happens for example when there is a large amount of records in a bulk.
